How C# works

C# is a compiled programming language, meaning the code you write is converted into machine language that the computer runs. If that doesn't make sense, then just think of it as if the computer speaks Arabic, and you only know English, so you write in English. Once you've finished your writing, the computer translates it into Arabic. Just imagine that, but instead of Arabic, it's binary.

That bit above isn't all that important. It's just some extra-fun theory.

When you write a C# program, in order for your code to run, you need to have:
  • A class (we'll cover that later) with the same name as the C# file,
  • and a function (or 'method') called Main (you can't change the name of it, otherwise it won't run by itself)
When you press the Run button on your code editor, or press F5, or whatever you do to run the code, C# (well, not actually the language itself but who cares) will look for a function called Main(), and will execute (run) the code inside it.

If you've used Unity, and are programming in vanilla C#, then the Main() is the equivalent of Start(), pretty much.

Variables

If you've programmed before, or been taught how to program, then you no doubt know what these are.
Especially if you learnt it from a school, then you'll also hate having to explain what it is. It's really weirdly difficult to explain. It's just a thing that you assign a value to, OK?

But, because this is a formal, no-nonsense tutorial, I have to formally explain what they are, with no nonsense at all.

A variable is an object that can be changed at runtime (when the program is running) that is given a name (identifier) and value.
A variable can be called anything, apart from reserved words, like if and void and using and other keywords like that.

Because this a C# tutorial, not an Arabic tutorial, we create variables like this: var variable = value;
As my Computing teacher once said, "your company will always give you a naming convention." But most of the time, you just call your variable myName. It's standard for most languages.

Datatypes

If you've come from JavaScript, or some other language that uses var, then you will recognise the word.
If you don't, and you've got here by looking up the best programming languages and realising that C# is quite high up, then all you might know is that var is short for variable.

There are lots of different 'datatypes' in C#. String, integer (whole number), float (decimal number), boolean (true/false), date/time, and so on.
I was deliberately using 'var' above to not confuse any new programmers. A new programmer would probably wonder what 'bool' is.
If that's you, it's absolutely fine. Bool is short for boolean.

If you want to create a variable with a specific datatype, you can write:
int number = 4;

Functions

You know the Main(string[] args) thing above? Let's try and remember the distant events of the mysterious past...

When you write a C# program, in order for your code to run, you need to have:
  • A class (we'll cover that later) with the same name as the C# file,
  • and a function (or 'method') called Main (you can't change the name of it, otherwise it won't run by itself)
I said that Main is a function.

But...
What is a function?

It's simple. Imagine you want to go to the shops and buy your food shopping.
You wouldn't want to repeat multiple steps over and over again.

For example, take when you start your car. Instead of performing the car-starting multiple times, you can just create a function, let's call it StartCar, and invoke (call) it multiple times. 
public class Example
{
	static void Main(string[] args)
	{
		StartCar();
		// Then go to the shops, and once you're finished:
		StartCar();
		// And drive home.
	}

	static void StartCar()
	{
		Console.WriteLine("The car is now on!");
		// Extra car-starting fun goes here!
	}
}
Instead of repeatedly running: 
Console.WriteLine("The car is now on!");
// Extra car-starting fun goes here!
you can just call StartCar() every time you want the car to start.

Now that the reason for creating functions is clearer, we can now dive deeper into the more advanced stuff.

Function parameters

Say you've got your car starting logic. Which you do.
How about actually doing the shopping? That would be useful for a shopping program.

In order to do that efficiently, you should use parameters. They are objects that you 'pass' to a function that change how it runs. It's like a variable, but each time you call the function, you give it a different value.

It's done like this: 
public class Example
{
	static void Main(string[] args)
	{
		BuyItem("Ice cream cone 4pk £3.49 on sale");
	}

	static void BuyItem(string itemName)
	{
		Console.WriteLine("You've bought " + itemName);
	}
}
The output would be:
> You've bought Ice cream cone 4pk £3.49 on sale

Warning:The parameter that you pass to the function is not accessible outside of it. You can not get the value of itemName when you're not in the BuyItem function.

Overloading

A lot of people would argue that overloading isn't necessary at this point, but it's a very useful thing to know, and since we're already covering functions, we may as well cover this as well.

Overloading is when you have two functions with the same name that take different parameters and can perform different actions.

If you want to have a BuyItem function that also takes an integer, holding the value of how many you want to buy, then you could just make a new function. Or, you could copy BuyItem and add an int parameter to it.

Observe: 
public class Example
{
	static void Main(string[] args)
	{
		BuyItem("Ice cream cone 4pk £3.49 on sale");
		BuyItem("Chocolate bar £0.49 on sale", 5); // Because they're on sale! Buy 5 of them!
	}

	static void BuyItem(string itemName)
	{
		Console.WriteLine("You've bought " + itemName);
	}
	
	static void BuyItem(string itemName, int amount)
	{
		Console.WriteLine("You've bought " + amount.ToString() + " of " + itemName);
	}
}
The output would be:
> You've bought Ice cream cone 4pk £3.49 on sale
> You've bought 5 of Chocolate bar £0.49 on sale

It's as simple as that. If that's not simple, then... well, it might not be, because I haven't explained it very well.
